# Профессиональный SEO-аудит greencard.agency (26 May 2026)

> Формат: коммерческий мультиязычный B2C-сайт страхового агента (пограничное автострахование).
> Метод: code-level аудит (Next.js App Router + словари + API), проверка генерации маршрутов/metadata/sitemap, анализ UX/конверсии.

## Контекст и ограничения проверки
- На момент проверки корневой домен `https://greencard.agency` отвечает `403` из окружения проверки (внешний live-crawl Lighthouse/CrUX недоступен).
- Поэтому выводы по indexability и CWV основаны на:
  - исходном коде проекта,
  - поведении `next build` (статическая генерация маршрутов),
  - реализации metadata/hreflang/sitemap/JSON-LD.

- Корректная i18n-архитектура с отдельными language URL (`/[lang]/...`) и централизованным `hreflang + x-default`.

- Есть юридические разделы (privacy/cookie/rules), что повышает trust baseline.

### Слабые стороны
- Слишком маленькая SEO-матрица URL для ниши cross-border страхования (фактически «витрина + ограниченное число лендингов»).
- Слабая глубина content-clusters под long-tail (страны/маршруты/тип ТС/срок/документы/проблемные сценарии).
- Недостаточно proof-контента для E-E-A-T (кейсы, публичные отзывы, процессные SLA, авторство экспертов, подтверждения лицензирования на уровне страниц).

### Критические ошибки
- **P0:** Историческая рассинхронизация `cookiepolicy` vs `cookie-policy` в sitemap/route. Сейчас в коде исправлено, но это контрольная зона регресса (нужен авточек в CI).
- **P1:** Нет системного контроля битых ссылок/orphan pages через регулярный рендер-краулинг и link graph.
- **P1:** Несоответствие SEO-амбиций и архитектурной глубины страниц (недостаточно посадочных для масштабного роста).

### Что снижает доверие Google
- Ограниченный объем «реальных» коммерческих сигналов по сравнению с лидерами рынка (объективные proof-пруфы, кейсы, детализация услуг по юрисдикциям).
- Недостаточная контентная гранулярность по интентам.

### Что снижает конверсию
- Для холодного трафика недостаточно upfront-доказательств надежности (сканы/реестры/кейсы/прозрачные SLA/понятные edge-cases по документам).
- Риск фрикции при файловой загрузке (частично смягчен логикой partial-success, но UX-предупреждений до отправки можно добавить).

## 2) Технический SEO-аудит

### 2.1 Индексация

#### robots.txt
- Генерируется через `app/robots.ts`, `allow: /`, `sitemap` и `host` заданы.
- Оценка: **хорошо**.

#### sitemap.xml
- Генерируется через `app/sitemap.ts` по `LOCALES × ROUTES`.
- В текущем коде маршрут cookie policy синхронизирован с роутингом (`/cookiepolicy`).
- Риск: нет test-guard на консистентность `ROUTES` vs `ROUTE_LAST_MODIFIED` keys.
- Приоритет: **P1** (добавить unit/check script).

#### canonical / hreflang / x-default
- Реализовано централизованно в `app/seo.ts` через `pageAlternates()`.
- `x-default` → английская версия (`/en...`), что логично для международного сценария.
- Оценка: **хорошо**, но нужен мониторинг Search Console по hreflang-errors.

#### Индексируемость / дубли / параметры / пагинация
- Пагинации и параметрических шаблонов практически нет (для текущего масштаба норм).
- Дубли между локалями допустимы в модели i18n, если корректно проставлен hreflang.
- Критичных index-bloat факторов в коде не выявлено.

#### Битые ссылки / orphan pages
- По коду видна ранее исправленная проблема sitemap mismatch.
- Полноценная проверка orphan невозможна без внешнего crawl (Screaming Frog/Sitebulb + логи).
- Приоритет: **P1** внедрить регулярный crawl-pipeline.

### 2.2 HTML и метатеги
- `title`/`description` заданы через словари для основных страниц.
- `H1-H2` структура есть на лендингах.
- Open Graph: базовые поля есть, но нет системного `og:image` пакета для всех страниц → **P2**.
- Twitter Cards отдельно не заданы → **P2**.
- `alt` по изображениям: визуальные активы ограничены; нужен чек coverage на всех `<img>` (для `next/image` тоже).
- JSON-LD: сильная часть проекта.
- Breadcrumb: JSON-LD есть; на SEO-лендингах добавлены видимые UI-крошки — это правильно.

### 2.3 Core Web Vitals / Perf

> Точных полевых LCP/INP/CLS нет (из-за отсутствия CrUX/live profiling из среды).

Ожидаемый профиль:
- TTFB: потенциально хороший из-за SSG.
- LCP/INP: риск деградации на mobile из-за тяжёлой интерактивной формы и клиентского JS.
- CLS: вероятно контролируемый, но без runtime-проверки не подтверждается.

Что может тормозить:
- Клиентская сложность в `LeadForm` (много полей, динамические блоки, файловые input).
- Возможная гидратационная стоимость на главной.

Что может ухудшать ранжирование:
- Если mobile INP/LCP выйдут за пороги — это снизит конкурентоспособность в SERP.

Next.js/React specific risks:
- **Hydration pressure** на форме заявки.
- **Metadata drift risk** при ручном расширении маршрутов без автопроверок.
- **Sitemap consistency risk** при несинхронном обновлении route maps.

Приоритеты:
- **P1:** замеры реальных CWV (RUM + GSC CWV report).
- **P1:** bundle-analyze, разделение тяжелых client-зон.
- **P2:** усилить preload/preconnect для критичных внешних источников (если используются).

## 3) SEO-архитектура

### Текущее состояние
- URL-структура: чистая, короткая, языковой префикс корректный.
- Вложенность: плоская, удобная для старта, но слабая для масштабного semantic coverage.
- Перелинковка: базовая; не формирует плотные topic hubs.

### Кластеры / каннибализация / long-tail
- SEO-кластеры есть лишь в зачатке (пара направленных посадочных).
- Каннибализация вероятна по mixed-intent ключам («green card / border insurance / OC graniczne») между общими страницами.
- Long-tail стратегия выражена слабо: мало страниц под конкретные маршруты/типы ТС/кейсы.

### Готовность к росту 500–5000 страниц
Сейчас: **не готов** (5/10).

Что нужно обязательно:
- Programmatic templates:
  1) `страна регистрации × страна въезда`;
  2) `тип ТС × срок полиса`;
  3) `документы/ошибки/штрафы/как купить на границе`;
  4) `стоимость/срок действия/исключения по странам`.
- Стандартизированные блоки для scale-publishing (FAQ, legality, CTA, proof).
- Автогенерация внутренних ссылок между смежными маршрутами.

Приоритет:
- **P0:** проектирование контентно-URL модели до запуска массового контента.
- **P1:** правила каноникализации/де-дупликации шаблонов.

## 4) Контентный SEO-аудит

### Качество и интент
- Базовый коммерческий интент закрыт.
- Недостаточная глубина экспертного контента для «сложных» сценариев клиента.
- Часть контента выглядит слишком универсально; слабый дифференциатор против крупных игроков.

### E-E-A-T и доверие
Плюсы:
- Юр. данные, контакты, структурированные бизнес-атрибуты в JSON-LD.

Минусы:
- Мало публично верифицируемых доказательств практики:
  - кейсы,
  - обезличенные примеры полисов/сценариев,
  - конкретные SLA/регламент обработки.

### Какие страницы слабые/перспективные
- Слабые: общие инфо-страницы без глубокой декомпозиции интентов.
- Перспективные: geo-лендинги и страницы под сценарии «что делать если...».

### Что расширять
- Разделы с country-specific правилами въезда и документами.
- Разделы с edge-cases: ошибка VIN, пограничные отказа, срочный полис, повторная оплата, неверный формат документов.

### Какие кластеры отсутствуют
- `border insurance [from-country] to [to-country]` в массовом покрытии.
- `стоимость + срок + тип ТС` отдельными URL.
- «Пошагово на границе» (чек-листы).

Приоритет:
- **P0:** приоритизация коммерческих long-tail кластеров.
- **P1:** редакционный календарь 3–6 месяцев.

## 5) Коммерческий SEO-аудит

### Конверсионность и UX
- Сильная сторона: короткий путь к лид-форме.
- Сильная сторона: мультиязычность снижает friction для международной аудитории.
- Риск: длинные формы + документы могут увеличивать drop-off на mobile.

### Trust/Leadgen
Что профессионально:
- Наличие правовых страниц.
- Наличие структурированной карточки компании.

Что выглядит слабее рынка:
- Недостаточно «доказательной упаковки» (отзывы, кейсы, процессные гарантии).
- Недостаточно явных аргументов «почему безопасно платить именно здесь».

Почему клиент может не оставить заявку
- Не уверен, что это официальный/надежный канал.
- Боится ошибки в документах/формате файлов.
- Не видит прозрачного SLA по времени реакции.

Где теряются лиды
- На этапе длинной формы.
- На этапе прикрепления файлов.
- На этапе неуверенности в легальности/гарантиях.

Приоритет:
- **P0:** блок доверия рядом с CTA (лицензии/регистрационные данные/процесс и SLA).
- **P1:** улучшить pre-upload UX (правила, примеры, автоматические подсказки).

## 8) SEO-потенциал

### Можно ли вывести в TOP Google
- Да, реалистично по long-tail и geo-комбинациям при системном расширении URL-карты.
- По широким head-term в лоб конкурировать будет сложно.

### Оценка объема
- Фаза 1: **120–250** коммерческих посадочных.
- Фаза 2: **80–180** информационных/FAQ/guide страниц.
- Дополнительно: шаблоны под сезонные/маршрутные паттерны.

### Сложность конкуренции
- Средне-высокая.
- Критический фактор — скорость контентного масштабирования + качество trust-сигналов.

Потенциал:
- Органический трафик: **средний → высокий** (при масштабировании).
- Лидогенерация: **средний → высокий** (если закрыть trust + UX узкие места).
- Перспектива 1–2 года: **позитивная**, если реализовать roadmap без пауз.

## 9) Что критично мешает продаже сайта как актива

Снижает стоимость:
- Небольшой текущий SEO-footprint (мало страниц/кластеров).
- Недостаточно публичных доказательств коммерческой зрелости.
- Ограниченная предсказуемость органического потока на текущем масштабе.

Выглядит недоделанным:
- Content architecture для high-scale growth.
- Система SEO-автоконтролей (regression checks по routes/sitemap/hreflang).

Что ожидает покупатель
- Подтвержденная динамика органики и лидов.
- Повторяемая масштабируемая модель генерации страниц.
- Прозрачный SEO/UX governance-процесс.

Что исправлять первым
- **P0:** масштаб URL+контент-модели под коммерческие long-tail сценарии.
- **P0:** усилить trust-блоки на ключевых посадках.
- **P1:** ввести технические SEO-чеки в CI/CD.

## Приоритизированный план работ

### P0 (0–30 дней)
1. Утвердить масштабируемую SEO-архитектуру URL (geo × vehicle × period × intent).
2. Внедрить trust-блоки на коммерческих страницах (лицензирование, SLA, доказательства процесса).
3. Запустить первую волну high-intent landing pages.

### P1 (1–3 месяца)
1. Настроить SEO regression checks: route↔sitemap consistency, hreflang coverage, noindex/canonical checks.
2. Внедрить регулярный рендер-краулинг (битые ссылки/orphan/redirect chains).
3. Сформировать и запустить editorial pipeline по информационным кластерам.

### P2 (3–6 месяцев)
1. Усилить social proof/кейсы.
2. Доработать OG/Twitter asset-пакет и мультиязычные креативы.
3. Построить dashboard по SEO+конверсии (landing-level).
