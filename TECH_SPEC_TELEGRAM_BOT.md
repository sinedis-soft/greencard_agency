# Техническое задание
## Проект: Telegram-экосистема Green Card Agency (бот + Mini App + операторский бот)

Дата: 26 мая 2026  
Версия: 1.0  
Основание: текущий сайт Green Card Agency (калькулятор, зона покрытия, форма заявки, мульти-язычность).

---

## 1. Цели и границы проекта

### 1.1 Цель
Разработать Telegram-решение на Python, включающее:
1. Клиентский Telegram-бот.
2. Встроенный калькулятор ориентировочной стоимости полиса.
3. FAQ-раздел.
4. Блок «Территория покрытия».
5. Приём заявок на страховку через Telegram Mini App (Web App внутри Telegram).
6. Перевод пользователя на оператора (через отдельный операторский бот и/или очередь операторов).
7. Интеграцию с Bitrix24 (контакт, компания, сделка, комментарии, вложения, статусы).

### 1.2 Бизнес-цели
- Повысить конверсию в заявку из мобильного канала Telegram.
- Сократить время первичного ответа клиенту.
- Централизовать лиды в Bitrix24.
- Снизить потери лидов за счёт обязательной валидации данных и маршрутизации к оператору.

### 1.3 Каналы взаимодействия
- Telegram Bot API (основной клиентский бот).
- Telegram Mini App (заявка в web-интерфейсе внутри Telegram).
- Operator Bot (второй бот для менеджеров/операторов).
- Backend API (Python) для бизнес-логики и интеграций.
- Bitrix24 REST API (CRM).

## 2. Рекомендуемая архитектура

## 2.1 Технологический стек
- **Python 3.12+**.
- Клиентский бот: **aiogram 3.x**.
- Backend API: **FastAPI**.
- Mini App frontend: React/Next.js или легковесный SPA (в зависимости от команды).
- БД: **PostgreSQL**.
- Кэш/очереди: **Redis** (FSM сессии, rate-limit, очереди операторов).
- Интеграции: HTTPX, Pydantic, Alembic.
- Фоновые задачи: Celery/RQ/Arq (по предпочтению команды).
- Логи/мониторинг: Sentry + Prometheus/Grafana (опционально).

### 2.2 Компоненты
1. **bot-client**: меню, FAQ, калькулятор, вход в Mini App, запрос оператора.
2. **mini-app**: форма заявки + валидации + загрузка файлов + отправка в backend.
3. **api-core**: единый слой бизнес-логики, расчётов, Bitrix24.
4. **bot-operator**: очередь заявок/чатов, принятие в работу, шаблоны ответов.
5. **integration-bitrix24**: маппинг полей и синхронизация статусов.
6. **notification-service**: уведомления пользователю и оператору о смене статусов.

### 2.3 Модель взаимодействия
1. Пользователь запускает client-бота.
2. Выбирает: калькулятор / FAQ / покрытие / оставить заявку / оператор.
3. Для заявки открывается Mini App внутри Telegram.
4. Mini App валидирует данные и отправляет в backend.
5. Backend создаёт/обновляет контакт/компанию/сделку в Bitrix24.
6. Backend уведомляет operator-бот о новой заявке.
7. Оператор берёт заявку в работу.
8. Пользователь получает автоуведомления о принятии/статусе.

## 3. Функциональные требования

### 3.1 Клиентский бот
Обязательные пункты меню:
- «🧮 Калькулятор»
- «❓ FAQ»
- «🌍 Территория покрытия»
- «📝 Оформить заявку» (открытие Mini App)
- «👨‍💼 Связаться с оператором»
- «🌐 Язык»

#### 3.1.1 Калькулятор
Входные параметры:
- Тип ТС: легковое / мото / автобус / грузовое / прицеп / спецтехника.
- Срок страхования: 30 / 60 / 90 / 180 / 365 дней.

Результат:
- Ориентировочная стоимость.
- Дисклеймер «Окончательная стоимость после проверки данных».

#### 3.1.2 FAQ
- Категории: покупка полиса, документы, сроки, оплата, возврат, покрытие, ошибки в данных.
- Поиск по FAQ (по ключевым словам).
- Кнопка «Не нашли ответ? Перевести на оператора».

#### 3.1.3 Территория покрытия
- Текстовый список стран покрытия.
- Визуальная карта (статичная картинка).
- Условия: типы ТС и границы сроков (мин 1 мес, макс 12 мес).

#### 3.1.4 Перевод на оператора
- Создаётся «диалог поддержки» с user_id и контекстом.
- Пользователь получает SLA-сообщение (например: «Оператор ответит в течение X минут»).
- При отсутствии онлайн-операторов — постановка в очередь.

### 3.2 Telegram Mini App (заявка)
- Запуск только из client-бота (через WebApp button).
- Авторизация пользователя через Telegram initData.
- Пошаговая форма (2 шага минимум: контакты и данные ТС).
- Поддержка нескольких ТС в одной заявке (опционально: 1 сделка = 1 ТС, но в рамках одной отправки).
- Загрузка фото/сканов документов (jpg/png/pdf).
- Чекбокс согласия с условиями и политикой.

### 3.3 Operator Bot
- Авторизация операторов по white-list Telegram ID.
- Очередь новых обращений.
- Статусы: New / In Progress / Waiting Client / Closed.
- Команды:
  - `/take <ticket_id>` — взять в работу
  - `/reply <ticket_id>` — ответ клиенту
  - `/close <ticket_id>` — закрыть
  - `/reassign <ticket_id>` — передать
- Быстрые шаблоны ответов.
- Лог действий оператора.

## 5. Поля Bitrix24 (контакт/компания/сделка)

> Рекомендуется использовать понятные кастомные коды полей (`UF_CRM_*`) и зафиксировать их в отдельном mapping.yaml.

### 5.1 Контакт (CRM Contact) — **реально используемые поля из текущего `app/api/lead/route.ts`**
Обязательные:
- `NAME` ← `first_name`
- `LAST_NAME` ← `last_name`
- `PHONE[WORK]` ← `phone`
- `EMAIL[WORK]` ← `email`
- `UF_CRM_1753957395750` ← язык клиента (через справочник `langToComm`)
- `UF_CRM_CONTACT_1686145698592` ← `passport_series_number`

Для физлица:
- `BIRTHDATE` ← `birth_date`
- `ADDRESS` ← `registration_address`

Важно:
- Поля Telegram (`telegram_user_id`, `telegram_username`) в текущем route **не пишутся в Bitrix24**.
- Поле `policyholder_type` в текущем route хранится логически (через `policyholder_isCompany`), но отдельным CRM-полем в Contact сейчас **не записывается**.

### 5.2 Компания (CRM Company) — при `policyholder_type=company` (реально в route)
- `TITLE` ← `Company {company_inn}` или `Company {last_name} {first_name}`
- `UF_CRM_COMPANY_1692911328252` ← `company_inn`
- `UF_CRM_1709019759168` ← `ceo_full_name`
- `UF_CRM_1709019814756` ← `ceo_title`

Важно:
- `company_country` и `company_legal_address` в текущем route в Bitrix24 **не передаются**.

### 5.3 Сделка (CRM Deal) — **реально используемые поля из route**
Базовые поля:
- `TITLE` ← `Lead {last_name} {first_name} {license_plate}`
- `CONTACT_ID`
- `COMPANY_ID`
- `COMMENTS` ← `comment`

Поля ТС и полиса:
- `UF_CRM_1686152306664` ← страна регистрации ТС (`countryFrom` через `countryMap`)
- `UF_CRM_1686152567597` ← тип ТС (`vehicleType` через `vehicleTypeMap`)
- `UF_CRM_1686152209741` ← период страхования (`period` через `periodMap`)
- `UF_CRM_1686152149204` ← дата начала (`startDate`)
- `UF_CRM_1686152485641` ← госномер (`plate`)
- `UF_CRM_1686152659867` ← VIN (`vin`)
- `UF_CRM_1686152515152` ← марка/модель (`brend`)
- `UF_CRM_1686152614718` ← год выпуска (`year`)
- `UF_CRM_1686152745455` ← тип двигателя (`engineType` через `engineTypeMap`)
- `UF_CRM_1686152831791` ← объём двигателя (`engineCapacity`)
- `UF_CRM_1686152861297` ← мощность (`vehiclePower`)
- `UF_CRM_1686152902186` ← единица мощности (`powerUnit` через `powerUnitMap`)
- `UF_CRM_1686154280439` ← документы (`docs`, файлы)

Служебные/дефолтные поля (как выставлено сейчас в коде):
- `UF_CRM_1693578066803` ← `"401"`
- `UF_CRM_1686682902533` ← `"813"`
- `UF_CRM_1690539097` ← `"425"`
- `UF_CRM_1700656576088` ← `""`
- `UF_CRM_1686683031442` ← `"231"`

Важно:
- `CATEGORY_ID`, `STAGE_ID`, `UF_CRM_REQUEST_ID`, `UF_CRM_CALC_PRICE_ESTIMATE`, `UTM_*`, `UF_CRM_TERMS_ACCEPTED`, `UF_CRM_PRIVACY_ACCEPTED`, операторские поля в текущем route **не передаются**.
- Если нужна Telegram-воронка и полноценная трассировка заявки, эти поля нужно добавить отдельно в API-контракт и в `crm.deal.add`.

## 6. Бизнес-логика

### 6.1 Общая логика client-бота
1. `/start` → приветствие + выбор языка.
2. Главное меню.
3. Любой сценарий должен иметь «Назад» и «Главное меню».
4. После 15 минут неактивности — мягкий reminder (1 раз).

### 6.2 Логика калькулятора
1. Пользователь выбирает тип ТС.
2. Выбирает срок.
3. Бот считает ориентировочную стоимость по матрице тарифов.
4. Вывод: цена + дисклеймер + CTA «Оформить заявку».
5. Сохранять событие аналитики `calc_completed`.

### 6.3 Логика FAQ
1. Показ категорий.
2. Выбор вопроса → ответ.
3. Оценка ответа (👍/👎).
4. При 👎 — предложить перевод на оператора.

### 6.4 Логика Mini App заявки
1. Вход в Mini App, проверка `initData`.
2. Заполнение контактов и данных страхователя.
3. Заполнение данных ТС и загрузка документов.
4. Проставление согласий.
5. Сабмит:
   - backend валидирует поля;
   - нормализует данные;
   - создает/обновляет Contact;
   - создает/обновляет Company (если юрлицо);
   - создает Deal(ы);
   - сохраняет файлы;
   - отдает `request_id` и статус.
6. Пользователь видит: «Заявка принята» + номер заявки.
7. В operator-бот уходит уведомление о новой заявке.

### 6.5 Логика оператора
1. Новая заявка попадает в очередь.
2. Первый свободный оператор берет заявку (`/take`).
3. Клиент получает сообщение «Оператор подключился».
4. Переписка логируется.
5. При отсутствии ответа клиента N часов — статус `waiting_client`.
6. После финализации — `closed`, отправка CSAT-опроса.

### 6.6 Логика ошибок и исключений
- Ошибка валидации → подсветка конкретного поля.
- Ошибка Bitrix24 → retry 3 раза с экспоненциальной паузой.
- Ошибка загрузки файла → повторная загрузка без потери формы.
- Если Bitrix недоступен > X минут — постановка заявок в локальную очередь + алерт администратору.

### 6.7 Антидубли
- Ключ дубля контакта: email/phone/telegram_user_id.
- Ключ дубля заявки: VIN + plate + start_date + phone (окно 24 часа).
- При дубле: новая сделка не создается, запись уходит в timeline существующей.

### 6.8 SLA
- Первичный автоответ: до 5 секунд.
- Создание лида в CRM: до 20 секунд.
- Время первого ответа оператора: целевое до 10 минут в рабочее время.

## 12. Критерии приемки

1. Пользователь может пройти сценарий от `/start` до отправки заявки без выхода из Telegram.
2. Все обязательные поля валидируются до отправки.
3. Заявка создается в Bitrix24 с корректным маппингом полей.
4. Оператор получает новое обращение и может ответить клиенту.
5. Переписка и статусы отражаются в CRM.
6. Ошибки интеграции не приводят к потере заявки (есть retry/очередь).
